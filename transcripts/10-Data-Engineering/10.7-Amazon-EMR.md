# 10.7 Amazon EMR (Elastic MapReduce)

## Qué es

- **Hadoop managed en AWS**
- Clusters de cientos de EC2 para big data
- Migración natural de Hadoop on-prem → AWS

## Ecosistema

- Apache Spark, HBase, Presto, Flink, Hive, etc.
- Provisioning y config automático
- Auto-scaling con CloudWatch

## Casos de uso

- Procesamiento de datos
- Machine Learning
- Web indexing
- Big data en general

## Arquitectura

- Se lanza en **una sola AZ** (mejor networking)
- EC2 con EBS (HDFS) para storage temporal
- **EMRFS** para storage permanente (S3)

```
EMR Cluster (single AZ)
├── EC2 + EBS (HDFS) ← temporal, high performance
└── EMRFS → S3 ← permanente, multi-AZ, durable
```

## Integraciones

- S3 via EMRFS
- DynamoDB via Apache Hive

## Tipos de Nodos

| Nodo | Función | Duración | Spot? |
|------|---------|----------|-------|
| **Master** | Coordina cluster | Long-running | ❌ On-Demand/Reserved |
| **Core** | Ejecuta + almacena | Long-running | ❌ On-Demand/Reserved |
| **Task** | Solo ejecuta | Opcional | ✅ Spot ideal |

## Opciones de compra

- **On-Demand**: confiable, nunca termina
- **Reserved**: ahorro, para Master/Core
- **Spot**: barato, para Task nodes

## Tipos de cluster

- **Long-running**: siempre activo, usar Reserved
- **Transient**: crear → procesar → destruir

## Configuración de instancias

### Uniform Instance Groups
- Un tipo de instancia por nodo
- Tiene auto-scaling

### Instance Fleets
- Mezcla de tipos y compra (On-Demand + Spot)
- Sin auto-scaling (por ahora)

## Para el examen

| Ves en pregunta | Respuesta |
|-----------------|-----------|
| Migrar Hadoop a AWS | EMR |
| Big data con Spark/Hive | EMR |
| Storage permanente en EMR | EMRFS (S3) |
| Storage temporal en EMR | HDFS (EBS) |

| EMR en múltiples AZ? | NO, single AZ |
