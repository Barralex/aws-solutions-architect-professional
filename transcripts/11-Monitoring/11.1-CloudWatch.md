# 11.1 - CloudWatch

## Transcript
CloudWatch deep dive covering Metrics, Alarms, and Synthetics Canary.

## Key Concepts

### CloudWatch Metrics
- AWS services provide metrics out of the box
- EC2 standard monitoring: **5 minutes** resolution
- EC2 detailed monitoring: **1 minute** resolution
- EC2 built-in metrics: CPU, Network, Disk (instance store only)
- **RAM is NOT a built-in metric** → need custom metric via CloudWatch Unified Agent
- Custom metrics: standard = 1 minute, high resolution = **1 second**

### CloudWatch Alarms - 3 Targets
1. **EC2 Actions**: Stop, Terminate, Reboot, Recover
2. **Auto Scaling Actions**: Scale in/out ASG
3. **SNS Notifications**: Email, fan out to SQS, trigger Lambda
- Bonus: Alarm events go to **EventBridge** → opens up Kinesis, Step Functions, Lambda, etc.

### CloudWatch Alarms - EC2 Recovery
- Monitor EC2 status check → trigger recover action
- Recovered instance keeps: private IP, public IP, EIP, metadata, placement group

### CloudWatch Dashboards
- Display metrics and alarms
- Can show metrics from **multiple regions**

### CloudWatch Synthetics Canary
- Scripts that monitor APIs, URLs, websites
- Reproduce what customers do programmatically
- Check availability and latency
- Store load time data and UI screenshots
- Written in **Node.js or Python**
- Uses headless Google Chrome
- Run once or on schedule

### Synthetics Canary Blueprints
1. **Heartbeat Monitor**: Load URL, store screenshot + HAR file
2. **API Canary**: Test read/write for REST APIs
3. **Broken Link Checker**: Validate all links in URLs
4. **Visual Monitoring**: Compare screenshots vs baseline
5. **Canary Recorder**: Record actions → generate script
6. **GUI Workflow Builder**: Test web forms (e.g., login)

### Synthetics Canary - Failover Use Case
CloudWatch Synthetics → detects issue → CloudWatch Alarm → Lambda → update Route 53 DNS → failover to healthy instance
