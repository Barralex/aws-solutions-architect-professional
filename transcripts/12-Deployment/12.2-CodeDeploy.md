# 12.2 - CodeDeploy

## Resumen
CodeDeploy despliega aplicaciones en EC2, ASG, Lambda y ECS. Servicio gestionado con diferentes estrategias según el target.

## Conceptos Clave

### CodeDeploy en EC2
- Agente CodeDeploy instalado en cada instancia
- Configuración en `appspec.yml`
- Actualizaciones **in-place** (no crea nuevas instancias)
- Hooks para verificar después de cada fase
- Estrategia ejemplo: "half at a time" (mitad a la vez)

### CodeDeploy en ASG
- **In-place**: actualiza instancias existentes + nuevas instancias creadas por scaling
- **Blue/Green**: crea nuevo ASG con instancias limpias, requiere ELB
  - Nuevas EC2 con v2, tráfico se mueve gradualmente
  - Instancias viejas se eliminan cuando todo OK

### CodeDeploy en Lambda
- Usa **traffic shifting del alias Lambda**
- Pre-traffic hook: Lambda que testea v2 antes de mover tráfico
- Post-traffic hook: Lambda que testea después de mover todo
- CloudWatch Alarm → rollback automático
- SAM usa CodeDeploy nativamente para desplegar Lambdas

### CodeDeploy en ECS/Fargate
- Blue/Green deployment
- Se configura en la **definición del servicio ECS** (no en consola CodeDeploy)
- Crea nuevo task set con nueva task definition
- Traffic shifting gradual via ALB
- Período de estabilidad antes de terminar tasks viejas
- Soporta Canary (ej: 10% por 5 min, luego 100%)

## Para el Examen
- EC2: in-place, agente necesario, appspec.yml
- ASG: in-place o blue/green (blue/green necesita ELB)
- Lambda: alias traffic shifting + hooks pre/post
- ECS: blue/green via task definitions, configurado en ECS service
- SAM deploys Lambda → usa CodeDeploy internamente
