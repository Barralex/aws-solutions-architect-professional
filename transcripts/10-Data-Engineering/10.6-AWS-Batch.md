# 10.6 AWS Batch

## Qué es

- Ejecutar **batch jobs** como contenedores Docker
- Fully managed (escala automático)
- Pago por recursos subyacentes (EC2/Fargate)

## Compute Options

1. **Fargate** - Serverless, sin provisioning
2. **EC2/Spot Instances** - Más control, más barato con Spot

## Triggers

- Lambda (SDK call)
- EventBridge (integración directa, sin código)
- Step Functions (orquestación)

## Arquitectura típica

```
S3 Upload → EventBridge/Lambda → Batch Job → S3 Output
                                    ↓
                               ECR (Docker image)
```

## Lambda vs Batch

| | Lambda | Batch |
|---|--------|-------|
| Tiempo máx | 15 min | Sin límite |
| Runtime | Built-in o imagen especial | Cualquier Docker |
| Disco | 10GB temporal | EBS/Instance store |
| Compute | Serverless fijo | EC2/Spot/Fargate |

## Compute Environments

### Managed (recomendado)
- AWS gestiona capacidad e instancias
- On-Demand o Spot (con precio máximo)
- Auto-scaling basado en cola de jobs
- Instancias en tu VPC (necesita NAT o VPC Endpoint para ECS)

### Unmanaged
- Vos gestionás instancias
- Más control, más responsabilidad

## Multi-Node Mode (HPC)

- Múltiples instancias para 1 job
- Main node + child nodes
- **NO funciona con Spot Instances**
- Mejor con Placement Group "cluster"

## Para el examen

| Ves en pregunta | Respuesta |
|-----------------|-----------|
| Job sin límite de tiempo | AWS Batch |
| Cualquier imagen Docker | AWS Batch |
| Job > 15 min | Batch (no Lambda) |
| HPC con múltiples nodos | Batch Multi-Node |
| Batch serverless | Batch con Fargate |
| Multi-Node + Spot | NO compatible |
